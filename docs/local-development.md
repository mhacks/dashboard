# Local development

## The rule

**Every branch develops against its own local database.**

Iterate with `db:push` (drizzle-kit push) for fast feedback, then generate a
migration file on your branch before opening a PR. The remote is never touched
during local development.

## Environment variables

Two git-ignored files at the repo root:

| File         | Purpose                                   | How to get it                                                                                         |
| ------------ | ----------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| `.env`       | Shared secrets (SMTP, remote credentials) | Copy from [Keys in Notion](https://app.notion.com/p/Keys-38124ca0c81b80ffac62f65acb442613)            |
| `.env.local` | Local Supabase + Storage connection       | Generated by `pnpm db:start` or `pnpm db:env` â€” see [Start the local stack](#1-start-the-local-stack) |

Day-to-day local development only needs `.env.local`. Root `.env` is for operations that
touch the remote (e.g. `supabase config push`) â€” see
[Remote development](./remote-development.md).

## 1. Start the local stack

`pnpm db:start` (= `supabase start` + env generation) boots a full Supabase stack in
Docker:

| Service  | URL                                    | Purpose                                   |
| -------- | -------------------------------------- | ----------------------------------------- |
| Postgres | `127.0.0.1:54322`                      | the database (`DATABASE_URL` points here) |
| API/Auth | `127.0.0.1:54321`                      | what the auth clients talk to             |
| Storage  | `127.0.0.1:54321/storage/v1/s3`        | S3-compatible API for resume uploads      |
| Studio   | `127.0.0.1:54323`                      | Supabase Studio â€” DB GUI                  |
| Mailpit  | `127.0.0.1:54324` (UI), `54325` (SMTP) | catches outgoing email locally            |

```bash
pnpm db:start       # boot Supabase in Docker + write .env.local
pnpm db:push        # apply the current schema to the fresh local db
pnpm dev            # run the app against localhost
```

Supabase Studio is available once the stack is up. [`supabase/config.toml`](../supabase/config.toml)
is the source of truth for auth rules, email templates, SMTP, `site_url`, and redirect
URLs â€” restart the stack after editing it (`pnpm db:stop` then `pnpm db:start`).

The base `[auth]` block in `config.toml` holds **local** values (e.g.
`http://127.0.0.1:3000`, SMTP disabled so Inbucket/Mailpit catches email). Production
overrides live in `[remotes.production]` and are only applied on `pnpm supabase config push` â€”
see [Remote development](./remote-development.md).

`db:start` runs [`scripts/gen-env-local.sh`](../scripts/gen-env-local.sh), which reads
`supabase status` and writes `.env.local` (git-ignored). Re-run
`pnpm db:env` if the stack is already up and you need fresh env values.

### Resume storage (local)

Resume uploads go through [`lib/aws/s3.ts`](../lib/aws/s3.ts). The same code runs in
every environment; `.env.local` sets `RESUMES_REGION=local` so the client uses
Supabase Storage instead of AWS. Production uses `us-east-2` â€” see
[Remote development](./remote-development.md).

**Bucket definition** â€” the local `resumes` bucket is declared in
[`supabase/seed.sql`](../supabase/seed.sql), **not** in `[storage.buckets.*]` blocks.
Keeping buckets out of `config.toml` prevents `supabase seed buckets --linked` from
provisioning them on the remote Supabase project (production uses AWS S3 instead).

**Bucket creation** â€” Supabase runs `seed.sql` automatically on first `supabase start`
and every `pnpm db:reset` (after migrations). No separate seed script is needed.

**Env wiring** â€” [`scripts/gen-env-local.sh`](../scripts/gen-env-local.sh) reads
`supabase status` and writes `.env.local`:

| Variable                    | Purpose                                  |
| --------------------------- | ---------------------------------------- |
| `RESUMES_REGION`            | `local` â€” selects Supabase Storage       |
| `RESUMES_S3_URL`            | local Storage URL from `supabase status` |
| `RESUMES_BUCKET`            | `resumes` (matches `seed.sql`)           |
| `RESUMES_ACCESS_KEY_ID`     | local Storage access key                 |
| `RESUMES_SECRET_ACCESS_KEY` | local Storage secret key                 |

To add another local bucket, add an `insert into storage.buckets` statement to
`supabase/seed.sql`, run `pnpm db:reset`, and update `RESUMES_BUCKET` in `.env.local`.

### App email (local)

Invite and role-change emails go through [`lib/aws/ses.ts`](../lib/aws/ses.ts) via
Nodemailer. The same code runs in every environment; `.env.local` sets `SMTP_HOST` so
the client uses Mailpit instead of AWS SES. Production uses `SES_ACCESS_KEY_ID` /
`SES_SECRET_ACCESS_KEY` â€” see [Remote development](./remote-development.md).

**Env wiring** â€” [`scripts/gen-env-local.sh`](../scripts/gen-env-local.sh) writes:

| Variable    | Purpose                         |
| ----------- | ------------------------------- |
| `SMTP_HOST` | `127.0.0.1` â€” selects Mailpit   |
| `SMTP_PORT` | `54325` â€” Supabase Mailpit SMTP |

View captured emails at `http://127.0.0.1:54324`.

## 2. Change the schema

The schema lives in [`lib/db/schema.ts`](../lib/db/schema.ts) â€” Drizzle owns every
table and column.

```bash
# edit lib/db/schema.ts
pnpm db:push        # drizzle-kit push â†’ your LOCAL db â€” iterate quickly
pnpm db:generate    # write the migration file for this change
```

Two conventions keep merges painless:

- **Keep changes additive** (new tables, nullable columns) until merged.
- **Generate the migration file on your branch** and commit it alongside
  `lib/db/schema.ts` before opening a PR.

## 3. Open a PR

```bash
git add lib/db/schema.ts supabase/migrations/
git commit -m "Add <thing> to schema"
git push -u origin <your-branch>   # then open a PR against main
```

CI runs Prettier, ESLint, and a production build â€” it never touches a database.

## Command reference

| Command            | Does                                                           |
| ------------------ | -------------------------------------------------------------- |
| `pnpm db:start`    | start Supabase (seeds on first boot), write `.env.local`       |
| `pnpm db:stop`     | stop the stack                                                 |
| `pnpm db:reset`    | wipe local Postgres, re-run migrations + `seed.sql`            |
| `pnpm db:env`      | (re)generate `.env.local` from the running stack               |
| `pnpm db:push`     | drizzle-kit push â€” apply `schema.ts` to the **local** db       |
| `pnpm db:generate` | drizzle-kit generate â€” write a migration from the schema delta |
| `pnpm db:migrate`  | drizzle-kit migrate â€” apply migrations to the **local** db     |

All `db:*` scripts target `127.0.0.1:54322` directly.
